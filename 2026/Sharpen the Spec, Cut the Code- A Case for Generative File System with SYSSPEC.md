SYSSPEC 的规约（Specification）设计是其核心创新，它摒弃了模糊的自然语言，转而采用一种受**形式化方法**启发的**结构化规约**。这种规约被设计为文件系统的“非歧义蓝图”，指导大语言模型（LLM）生成准确且可进化的代码。

其规约设计主要由以下三个维度组成：

### 1. 功能规约 (Functionality Specification)
功能规约通过描述**状态转换**来定义模块的行为，包含四个关键组件：
*   **前置/后置条件 (Pre- and Post-conditions)**：借鉴 **Hoare 逻辑**，定义函数执行前必须满足的状态以及执行完成后保证达到的状态。例如，写入操作的后置条件会明确规定文件大小等于 `max(旧大小, 偏移量 + 长度)`，而不是含糊地描述“更新文件大小”。
*   **不变量 (Invariants)**：定义在所有状态转换中都必须保持成立的属性，例如“root 节点永远存在”或“修改 inode 前必须持有对应的锁”。
*   **意图 (Intent)**：用自然语言描述的高级目标，辅助 LLM 理解设计意图并注入特定领域知识。
*   **系统算法 (System Algorithm)**：为了避免 LLM 生成正确但低效的代码（如使用冒泡排序而非快速排序），开发者可以显式概述算法逻辑，确保性能表现。

### 2. 模块化规约 (Modularity Specification)
为了应对文件系统复杂的组件组合，SYSSPEC 引入了**依赖-保证（Rely-Guarantee）**机制，将系统解耦为独立且可组合的模块：
*   **依赖 (Rely)**：枚举该模块对其他组件的假设（如依赖的结构定义、锁定原语、路径遍历函数等）。
*   **保证 (Guarantee)**：定义该模块对外导出的接口合同，即它承诺向其他模块提供的行为。
*   **上下文约束 (Context-bounded)**：每个模块的规约和实现被限制在 LLM 的**上下文窗口**之内（例如 ≤500 行代码），以确保模型能够进行全局性的整体分析。

### 3. 并发规约 (Concurrency Specification)
这是 SYSSPEC 处理文件系统复杂性的关键。由于 LLM 难以直接生成复杂的并发逻辑，SYSSPEC 采取了**解耦策略**：
*   **独立设计**：将锁定协议、顺序要求等并发行为从功能逻辑中剥离，形成独立的并发规约。
*   **两阶段生成 (Two-phase Generation)**：LLM 首先根据功能规约生成**顺序逻辑代码**；在通过验证后，再根据并发规约在代码中**植入锁操作**和其他并发控制机制。
*   这种方法使得 LLM 能够一次专注于一个复杂问题，从而显著降低死锁和竞争风险。

### 4. 语言风格与精度
为了平衡**可理解性**与**精确性**，SYSSPEC 的规约并非纯数学逻辑，而是采用**增强的结构化自然语言**：
*   使用**类型注解**和**数学规范化**的自然语言表达式（如上文提到的文件大小计算公式）。
*   通过严格的章节组织（功能、模块化、并发）来消除歧义，确保 LLM 和人类开发者都能清晰理解设计蓝图。

这种规约设计的优势在于，它将开发者的负担从编写底层 C 代码转向高级设计，使得系统更易于维护，并能通过“规约补丁（Spec Patch）”实现高效的功能演进。
